## 토론 1: % 수치 팩트 정리

### 현황 파악 결과

**신고가 `priceChangeRate`**
- 구조: `getRecordHighs`에서 이번 달 최고가 vs 전월 최고가 비교로 실 계산 가능
- 현재 상태: 실 API 키 없음 → Mock fallback (`RECORD_HIGH_MOCK` 하드코딩, 5.7%/2.6% 등)
- 사용자 화면: 기준 설명 없이 "+5.7%" 표시 중

**HOT `tradeSurgeRate`**
- 구조: `types/index.ts:361`에 타입 정의만 존재
- `HOT_APARTMENTS_MOCK`에 해당 필드 없음 (Mock에서 생성 안 됨)
- 실 API 경로 `getHotApartments`에서도 전월 비교 로직 없음 — 계산 불가
- 사용자 화면: `tradeSurgeRate > 0` 조건이라 대부분 미표시. 간헐적 undefined 표시 위험

### 논의 과정

- 동현: "사용자는 이 숫자가 Mock인지 모른다. 신뢰성 문제."
- 하늘: "신고가 % 숫자만의 문제가 아니라 Mock 데이터를 실 데이터인 척 서빙하는 구조 자체가 문제."
- 재훈: "삭제 or 실 계산 교체. 기준 명시는 실 데이터 연동 후."
- 지수: "% 제거 시 정보 밀도 저하 — 절대 금액 차이(+X억)로 교체하면 직관적이고 실 데이터 경로에서 이미 `previousPrice` 존재."
- 서연: "신고가 `previousPrice` Mock에 추가 가능. `recentPrice - previousPrice` 절대값 계산 즉시 가능."

### 합의

- 신고가 `priceChangeRate` % 뱃지 제거 → **절대 금액 차이 (+X억) 표시**로 교체
  - BE: Mock `previousPrice` 필드 추가 (서연)
  - FE: RecordHighCard UI 교체 (민준)
- HOT `tradeSurgeRate` 뱃지 **전면 제거** → **`tradeCount` N건 표시**로 교체
  - FE: `tradeSurgeRate > 0` 블록 제거 + `tradeCount` 표시 (민준)

## 토론 2: 사용자 여정 분석

### 시나리오 A — 서울 아파트 사러 온 30대

**현재 여정:** 홈 진입 → 검색(헤더) → 단지 상세 → PriceChart
**문제점:**
- 지도 배너가 홈 맨 아래 위치. 지역 탐색하려는 사람이 스크롤 끝까지 가야 발견
- 홈 콘텐츠 순서(MarketBanner → 신고가 → 청약 → HOT → 지도)가 이 시나리오의 우선순위와 불일치
- PriceChart 기본 표시 기간 불명확 (사용자가 3개월인지 1년인지 모름)

**결론:** 지도 배너 위치 재검토 필요. 섹션 순서 개편 시 이 여정을 우선 고려.

### 시나리오 B — 청약 당첨 노리는 사람

**현재 여정:** 홈 진입 → WeeklySubscriptionTimeline → 날짜 클릭 → 카드 펼침 → **막힘**
**문제점:**
- WeeklySubscriptionTimeline 카드 클릭 → onClick 없음. 종착지 없음
- 청약 상세 페이지(`/subscriptions/:id`) 미착수 (PRD P1)
- 사용자가 청약 카드에서 더 알고 싶어도 갈 곳이 없음

**결론:** 카드에 임시 CTA `/subscriptions` 라우팅 추가. 청약 상세 페이지는 P1 우선순위.

### 시나리오 C — 내가 사는 아파트 시세 궁금한 사람

**현재 여정:** 홈 진입 → 검색 → 단지 상세 → PriceChart
**문제점:**
- 검색 결과가 Mock 데이터 기반 → 실제 단지 검색 안 될 수 있음
- BookmarkUpdateSection이 이 시나리오에 가장 유용하지만 북마크 없는 첫 방문자에게 미노출

**결론:** 검색 실 데이터 연동이 핵심. BookmarkUpdateSection은 북마크 유도 온보딩과 연결 필요.

## 토론 4: 봄집 차별화 포인트

### 현재 상태 평가

- 호갱노노 대비 현재 봄집은 기능 10% 수준 (솔직한 팀 합의)
- 실 데이터 연동 없이 차별화 주장 불가 (동현, 재훈 동의)

### 잠정 결론

**단기 포지셔닝:** "청약 탐색 → 실거래가 확인을 하나의 여정에서 제공"
- 호갱노노: 실거래가 지도 강점. 청약은 별도 서비스
- 아실: 데이터 분석 강점. 청약 정보 부족
- **봄집:** 청약 + 실거래가 통합 여정. 처음 집 보는 30대 타겟 온보딩 UX

**중기 방향:** 처음 집 사는 사람을 위한 단계별 가이드 UX — 정보 과부하 없이 진입 장벽 낮춤

**전제 조건:** 말이 실행이 되려면 청약 상세 페이지 + 실 API 연동이 선행.

## 토론 5: 정보 신뢰성 현황 점검

### 홈 섹션별 데이터 실태

| 섹션 | 데이터 종류 | 비고 |
|------|----------|------|
| WeeklySubscriptionTimeline | 실 API (청약홈) | 정상 |
| BookmarkUpdateSection | localStorage | Mock 없음 |
| MarketSummaryBanner | Mock | trend.service.ts — 실 기반 아님 |
| RecordHighSection | Mock fallback | API 키 있으면 실 계산 가능 |
| HotApartmentSection | Mock fallback | API 키 있으면 실 거래량 기준 |
